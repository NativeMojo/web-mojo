# Examples Portal — multi-sidebar reorganization

| Field | Value |
|-------|-------|
| Type | request |
| Status | planned |
| Date | 2026-04-26 |
| Priority | medium |

## Description

Replace the single 75-item sidebar in `examples/portal` with a **hub-and-spoke set of topic sidebars**.

The default sidebar becomes a small "Examples Hub" with 6–7 large topic entries. Clicking a topic switches to a focused sub-sidebar that contains only the examples for that topic, plus a "Back to Hub" item at the bottom (mirroring the existing "Exit Admin" pattern). Subroutes for the same component (e.g. `components/dialog`, `components/dialog/form`, …) collapse under their parent as nested children.

No framework changes are required — the `Sidebar` component already supports multiple registered menus, route-driven menu switching, per-item `handler` callbacks, and one-level nested children. This is a config + registry refactor.

## Context

`examples.registry.json` currently produces a single sidebar with **75 items in 8 sections**:

| Section      | Items |
|--------------|-------|
| Components   | 22    |
| Forms        | 14    |
| Core         | 11    |
| Extensions   | 11    |
| Form Inputs  | 7     |
| Pages        | 5     |
| Services     | 3     |
| Models       | 1     |

Sections render as collapsible parent items, but every variant subroute (4 Dialogs, 3 Modals, 4 TableViews, 3 ListViews, 3 TablePages, 2 ContextMenus, 2 FileViews, 2 TabViews, 3 Charts, 2 Validations, 2 FormViews) is a sibling of its parent — so the visible flat list is much longer than the 8 collapsed sections suggest. The result is a wall of links that hides the canonical learning path and is hostile to both developers skimming for a feature and LLMs scanning for the right reference.

The `system` (admin) sidebar already proves the pattern works: a topbar action calls `app.sidebar.setActiveMenu('system')`, a "Exit Admin" handler item at the bottom switches back. We extend that pattern.

## Acceptance Criteria

- [ ] Default sidebar ("hub") shows ≤ 8 entries: Home, the 5–6 topic links, plus a divider and Admin shortcut.
- [ ] Clicking a topic entry switches the sidebar to the corresponding sub-menu (no route change required for the switch itself; the topic items inside the sub-menu carry their own `?page=` routes).
- [ ] Every sub-menu ends with a `spacer` + a "Back to Examples" item that calls `app.sidebar.setActiveMenu('hub')` — same shape as today's `exit-admin` item.
- [ ] Sub-menus group component variants under one collapsible parent item (e.g. one "Dialog" item with 4 children, one "TableView" with 4 children) instead of 4–5 sibling rows.
- [ ] Routes do not change. Existing URLs (`?page=components/dialog/form`, `?page=forms/inputs/tag-input`, etc.) keep working. The HomePage card grid keeps working.
- [ ] When the user lands on a deep route (e.g. `?page=components/table-view/batch-actions` from a bookmark or LLM), the sidebar auto-switches to the sub-menu that contains that route — `Sidebar.autoSwitchToMenuForRoute` already does this; we just need each route to be reachable from exactly one menu.
- [ ] The hub menu visually communicates "you are in a topic" (sub-sidebar uses a distinct `className` like the admin menu's `sidebar-light sidebar-admin`, with a small header showing the topic name).
- [ ] `examples.registry.json` is regenerated by `scripts/build-registry.js` and remains the single source of truth for both the sidebar and `docs/web-mojo/examples.md`.
- [ ] HomePage continues to list every example grouped by area (no behavior change there).

## Investigation

### What exists

- `examples/portal/app.js` — builds `sidebarItems` from `registry.menu` and registers two sidebar menus (`default`, `system`). The `system` menu has a `handler` item that calls `app.sidebar.setActiveMenu('default')` to exit. Topbar `rightItems` includes a wrench shortcut that switches to `system` via `setActiveMenu`.
- `examples/portal/examples.registry.json` — 75 pages, plus a `menu` array with 8 area objects (`{ area, section, icon, pages: [...] }`). Generated by `scripts/build-registry.js` from each example's `example.json`.
- `examples/portal/scripts/build-registry.js` — walks `examples/portal/examples/<area>/<Component>/example.json`, validates, sorts, and emits `examples.registry.json` + `docs/web-mojo/examples.md`. Has a fixed `AREA_ORDER` and `AREA_LABEL` map. Does **not** know about menu-grouping for sub-sidebars.
- `src/core/views/navigation/Sidebar.js` — supports `addMenu(name, config)`, `setActiveMenu(name)`, `defaultMenu`, route → menu auto-switch, one level of nested `children`, and per-item `handler` callbacks. No framework change required.

### What changes

- `examples/portal/app.js` — replace the single `sidebarItems` build loop with one builder per topic menu, register all six menus on the PortalApp, register a topbar action handler per topic (or add an "open-topic" generic action keyed off the menu name).
- `examples/portal/examples.registry.json` — add a `topics` array (or rename `menu` to `topics`) keyed by topic-menu name, where each topic specifies its `groups` (collapsible parent items) and which routes go in each group. The existing flat `pages` array stays (HomePage uses it).
- `examples/portal/scripts/build-registry.js` — extend with a topic configuration. Two viable shapes:
  1. **Static map in the script**: `TOPIC_MAP = { 'components': { groups: [{ label: 'Modals & Dialogs', match: ['Dialog', 'Modal'] }, … ] } }`. Build-registry computes the topic+group for each page from this map. Lowest churn, all the design lives in one file.
  2. **Per-example manifest field**: add `"topic": "components"` and `"group": "Modals & Dialogs"` to each `example.json`. More work to author, more flexible long-term. Recommend (1) for now — 75 entries, finite topics, one place to edit.
- `examples/portal/shell/HomePage.js` — no required change. Optional: add the topic name as a chip on each card.

### Constraints

- **No framework changes.** The Sidebar already handles everything we need. If we discover a missing capability (e.g. nested grandchildren), open a separate request — do not change `src/core/views/navigation/Sidebar.js` as part of this.
- **Routes are immutable.** Existing `?page=…` URLs must keep resolving to the same example. Bookmarks, deep links, and the LLM-facing `find-example` skill all index off `route`.
- **Registry stays the source of truth.** `app.js` should read whatever the build emits; do not hand-edit menu items in `app.js` outside the `hub` shell itself.
- **Build determinism.** `build-registry.js` already emits a sorted, byte-identical file. The new topic map must preserve that — sort groups by configured order, pages within a group by `order` then `title`.
- **Permissions / auth.** This is the public examples portal with auth disabled and `hasPermission = () => true`. No permission gating on topic items is required.

- Manual smoke: load the portal, click each hub entry, verify the sub-sidebar opens with the expected groups, click "Back to Examples" returns to the hub, deep-link to a sub-route (e.g. `?page=components/dialog/form`) and verify the sidebar auto-switches to `components` with Dialog expanded.

### Out of scope

- No tabbed-page consolidation of variants (Dialog basic / form / context-menu / custom-body stay as separate routes — only the sidebar nests them).
- No new sidebar search / filter input. (Possible follow-up; a small per-menu list is the goal of this work.)
- No changes to the admin `system` menu.
- No changes to `HomePage` other than optional cosmetic topic chips.
- No changes to the `find-example` skill or `docs/web-mojo/examples.md` shape beyond what `build-registry` regenerates automatically.

## Plan

### Objective

Reorganize the Examples Portal into a hub-and-spoke navigation model with two coordinated surfaces:
1. **Sidebar (human)** — small hub menu + four topic sub-sidebars (Architecture, Components, Forms, Extensions), each with a "Back to Examples" footer. Existing `system` admin menu unchanged. Component variants collapse one level under a parent. Group section labels are non-clickable.
2. **Registry (LLM)** — `examples.registry.json` gains `topic` + `group` per page and a `topics` tree. `docs/web-mojo/examples.md` regenerates under the new taxonomy. No routes change. No framework changes.

Hub pins a curated **Start Here** path: View, Templates, Model, Page, WebApp.

### Coverage check (77 examples, all accounted for)

- **Architecture (20)**: Core 11 + Pages 5 + Services 3 + Models 1
- **Components (23)**: Modals & Dialogs 7 (Dialog+3, Modal+2) · Lists & Tables 8 (ListView+2, TableView+3, DataView) · Files 3 (FileView+1, ImageFields) · Navigation 4 (Sidebar&TopNav, SideNavView, ContextMenu+1) · Other 1 (ChatView)
- **Forms (21)**: FormView 2 (FormView+1) · Field Types 7 · Specialized Inputs 7 · Patterns 5 (Validation+1, Layout, Wizard, Search&Filter)
- **Extensions (13)**: Charts 5 (MiniChart, SeriesChart, PieChart, CircularProgress, MetricsMiniChartWidget) · Maps & Location 3 · Media 2 · UI 3 (TabView+1, TimelineView)

### Steps

1. **`examples/portal/scripts/build-registry.js`** — add `TOPIC_TAXONOMY` constant (single source of truth listing every topic, group, component, and which routes are children of which parent). Populate `topic` + `group` on every page during `expandManifest`. Add `buildTopics(pages)` producing the new `topics` tree. Update `buildDocsIndex` to render H2/H3 by topic/group and link each row to its actual `sourcePath` (incidental fix to the variant-link bug). Fail fast if a page is missing from the taxonomy.
2. **`examples/portal/examples.registry.json`** — regenerated. Adds `topics` tree alongside the existing `pages` and (legacy) `menu`. `pageCount` stays in sync.
3. **`examples/portal/app.js`** — replace single sidebar build with: hand-built `hub` menu (Home, Start Here strip, four topic links, Admin) + a `buildTopicMenu(topic)` helper that consumes `registry.topics` and emits sidebar config (header, `kind:'label'` group headers, item leaves with optional one-level children, spacer, "Back to Examples" item). Register all menus with `defaultMenu: 'hub'`. Keep `system` admin menu untouched. Keep wrench-icon shortcut (defensive `hasMenu` guard before `setActiveMenu('system')`).
4. **`examples/portal/shell/HomePage.js`** — switch to render `topics` (Start Here strip on top, then per-topic / per-group sections of cards). Fall back to legacy `areas` if `topics` not provided.
5. **`examples/portal/app.css`** — small `.sidebar-topic` and `.sidebar-section-label` blocks (~15 lines).
6. **`test/build/examples-registry.test.js`** — extend with: every page has non-empty `topic`/`group`; every flat-page route appears in exactly one place inside `topics`; every topic has a unique name and at least one group; the five Start Here routes resolve to existing pages.
7. **`examples/portal/README.md`** — one paragraph on the topic taxonomy and how to extend it.

### Design Decisions

- **Static taxonomy in build script, not per-example fields** — 75+ manifests would need editing for the alternative. Information architecture belongs in one file.
- **Keep legacy `menu` array in registry one cycle** — existing tests assert it; remove in a follow-up.
- **Start Here inlined in hub, not its own sub-sidebar** — fewest clicks to the canonical onboarding pages.
- **`kind: 'label'` for group headers** — Sidebar template only recurses one level; nested grandchildren would need a framework change (forbidden by constraints).
- **Fix `examples.md` variant-link bug as part of this work** — `buildDocsIndex` is being rewritten anyway; leaving the bug in new output would be worse.

### Edge Cases

- Deep-link first-load (`?page=components/dialog/form`): `Sidebar.autoSwitchToMenuForRoute` walks all menus and switches to the topic containing the route; `setActiveItemByRoute` activates the parent + child so the collapse opens automatically.
- New example without taxonomy entry: build fails with a clear message.
- Empty group after filtering: not emitted; build script logs a warning.
- Single-page component: emits as flat leaf (no `children` array).
- `system` menu registered late by `mountAdminExtension()` (post `app.start()` setTimeout): hub Admin handler and wrench handler both `if (!app.sidebar.hasMenu('system')) return;` guard.
- HomePage receives no `topics`: falls back to legacy `areas` rendering.
- Determinism: taxonomy order drives sort; existing byte-identical-regen test still passes.

### Testing

- `node examples/portal/scripts/build-registry.js` exits 0; regenerates registry and `docs/web-mojo/examples.md`.
- `npm run test:build` green (existing tests + new assertions).
- Manual smoke: hub renders → click Components → see groups → click Dialog → expands to 3 children → click "Back to Examples" → returns to hub → deep-link `?page=components/dialog/form` opens Components with Dialog expanded → wrench icon and hub Admin both reach the admin sidebar → home page mirrors the topic taxonomy.

### Docs Impact

- `docs/web-mojo/examples.md` regenerated under the new H2/H3 topic structure with correct per-variant source links.
- `examples/portal/README.md` gets one paragraph on the taxonomy.
- `CHANGELOG.md` entry under Examples Portal.
- No changes to framework docs.

### Out of Scope

- No changes to `Sidebar.js` or any framework component.
- No tabbed-page consolidation of variants.
- No sidebar search box.
- No changes to per-example `example.json` manifests.
- No changes to the `system` admin menu items.

## Resolution

**Status**: Resolved — 2026-04-26

### What was implemented

Hub-and-spoke navigation for the Examples Portal. Single 77-item sidebar replaced with:

- **Hub menu** (10 items): Home + 5 pinned "Start Here" learning-path links + 4 topic links + Admin.
- **4 topic sub-sidebars**: Architecture (20), Components (23), Forms (21), Extensions (13). Each has section labels for groups, collapsible parents for component variants, and a "Back to Examples" footer.
- **Admin sidebar** unchanged.

`examples.registry.json` now carries `topic` and `group` on every page plus a top-level `topics` tree that the LLM and the sidebar both read from. The legacy `menu` array is kept one cycle. `docs/web-mojo/examples.md` is regenerated under H2/H3 topic/group headings with correct per-variant links.

Sidebar widened to 300px via `--mojo-sidebar-width` (longest titles like "TableView — server collection" now fit cleanly).

No framework changes.

### Files changed

- `examples/portal/scripts/build-registry.js` — added `TOPIC_TAXONOMY` constant, `buildTopics()`, route-coverage validation, rewrote `buildDocsIndex()`.
- `examples/portal/examples.registry.json` — regenerated; gains `topics` tree and per-page `topic`/`group`.
- `examples/portal/app.js` — replaced single sidebar build with hub + `buildTopicMenu()`; added `switchToMenu` guard for the late-registered `system` menu.
- `examples/portal/shell/HomePage.js` — renders `topics` (Start Here strip + per-topic / per-group cards).
- `examples/portal/app.css` — added `--mojo-sidebar-width: 300px`, `.sidebar-topic`, `.sidebar-section-label` styles.
- `examples/portal/README.md` — documented the topic taxonomy.
- `test/build/examples-registry.test.js` — added 4 new assertions.
- `docs/web-mojo/examples.md` — regenerated.
- `CHANGELOG.md` — added entry under Unreleased.

### Tests run

- `node test/test-runner.js --suite build` — all 14 examples-registry assertions pass (10 existing + 4 new). The 7 unrelated build-verification failures (missing `dist/index.html`, source maps, app.json) pre-date this change and require `npm run build` first.
- `npm run lint` — no new errors. All 16 errors pre-existed in `src/core/*` files I did not touch.
- Manual browser verification on the running dev server:
  - Hub renders with all 10 expected items (Home, START HERE label, View/Templates/Model/Page/WebApp, BROWSE label, 4 topic links, Admin).
  - Switching to Components shows 5 group labels (Modals & Dialogs, Lists & Tables, Files, Navigation, Other).
  - Dialog parent is collapsible with 3 children rendered inside `<div class="collapse">`.
  - Deep-link to `?page=components/dialog/form` auto-switches sidebar to Components, expands Dialog, highlights "Dialog — forms".
  - "Back to Examples" returns to hub.
  - Sidebar width 300px on desktop viewport.

### Validation

77 examples mapped 1:1 into the topic taxonomy (verified by build-time `TOPIC_TAXONOMY` validation: every registered route must appear in the taxonomy, and every taxonomy route must resolve to a registered page; otherwise the build fails). Routes did not change. The hub's curated Start Here routes are asserted to resolve to existing pages by the new test.
